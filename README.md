# TwentyOz 개발자 채용 과제 — 사용자/권한/메뉴 관리

안녕하세요. 지원해 주셔서 **감사합니다.**

이 과제는 저희가 실제 업무에서 사용하는 스택과 도메인으로 구성했습니다. 화려한 알고리즘 문제가 아니라, **입사 후 실제로 맡게 될 "관리자 시스템"을 작게 만들어 보며** 서로의 일하는 방식을 확인하기 위한 과제입니다. 그래서 단일 테이블 CRUD가 아니라 **로그인 → 권한 → 메뉴**가 하나의 흐름으로 연결되도록 설계했습니다.

저희가 보는 것은 완벽한 결과물이 아니라, 요구사항을 정확히 이해하고 **일관된 컨벤션으로 동작하는 코드**를 만드는 능력입니다. 진행하다 막히거나 요구사항이 모호하면 **혼자 끙끙 앓지 말고 [오픈채팅방](#4-문의--실시간-질문방-)에서 편하게 물어보세요.** 그래도 본인 판단으로 정한 부분이 있다면, 제출 시 **PR 양식**에 그 가정과 이유를 적어 주시면 됩니다.

## 과제 안내

먼저 아래 다섯 줄로 전체 그림을 잡고, 자세한 내용은 각 장에서 확인하세요.

| 항목 | 요약 |
|---|---|
| **무엇을** | 사용자 관리 · 권한 관리 · 메뉴 관리 + 로그인/인증 → [1장](#1-무엇을-만드나요) |
| **스택** | NestJS + React (이 레포가 turbo 모노레포 스타터) → [3장](#3-기술-스택--스타터) |

## 1. 무엇을 만드나요

세 가지 화면과 그것을 묶는 로그인/인증을 만들어 주세요.

### (0) 로그인 / 인증
- **ID(또는 이메일) + 비밀번호** 로그인 페이지
- 로그인 성공 시 **JWT를 발급**하고, 이후 요청에서 토큰을 검증해 보호 라우트에 접근
- 비로그인 상태로 관리 화면 접근 시 로그인 페이지로 리다이렉트

### (1) 사용자 관리 (User)
- 사용자 **목록 / 등록 / 수정 / 삭제**
- 각 사용자에게 **역할(Role)** 부여 (예: `ADMIN`, `USER`)
- 비밀번호는 평문 저장 ❌ — 해시(예: `bcrypt`)로 저장

### (2) 권한 관리 (Role)
- **역할(Role) 기반의 RBAC**를 구현합니다. (RBAC = Role-Based Access Control — 사용자에게 직접 권한을 주는 대신, **역할에 권한을 묶고 사용자에게 역할을 부여**하는 방식)
- 역할 **목록 / 등록 / 수정 / 삭제** (최소한 `ADMIN` / `USER`는 시드로 제공)
- **각 역할이 어떤 메뉴에 접근 가능한지**를 지정 (역할 ↔ 메뉴 매핑)

### (3) 메뉴 관리 (Menu)
- 메뉴 **목록 / 등록 / 수정 / 삭제**
- 로그인한 사용자의 **역할에 따라 사이드바(또는 내비게이션) 메뉴가 동적으로 그려지도록** 해 주세요.
  - 예) `USER`로 로그인하면 "사용자 관리"·"권한 관리"가 안 보이고, `ADMIN`이면 전부 보임

> 💡 백엔드 API와 프론트엔드 화면을 **둘 다** 만들어 주세요. 한쪽만 가능하다면 가능한 쪽을 완성하고 README에 그 사실을 적어 주세요.

## 2. 핵심 흐름

```
[로그인]  ID/PW 입력
   │
   ▼  인증 성공
[JWT 발급]  토큰 안에 userId, role 포함
   │
   ▼  이후 모든 요청에 토큰 첨부
[보호 라우트]  토큰 검증 → 통과 시 접근
   │
   ▼  로그인 사용자의 role 조회
[메뉴 렌더링]  role이 접근 가능한 메뉴만 사이드바에 표시
```

메뉴 필터링을 프론트에서 할지 백엔드에서 내려줄지, 토큰에 role을 담을지 매번 조회할지 등 — 정답은 없으니, **어떤 방식을 골랐고 왜 그렇게 했는지**를 제출 시 PR 양식에 적어 주세요.

> 💡 권한 판단은 **신뢰할 수 있는 위치(서버)**에서 이뤄지도록 해 주세요. 프론트에서 메뉴를 숨기는 것만으로는 API가 무방비가 됩니다.

## 3. 기술 스택 & 스타터

저희가 실제로 쓰는 스택입니다. (버전은 권장값이며, major가 같으면 무방합니다.)

> 🚀 **이 레포 자체가 turbo 모노레포 스타터입니다.** `apps/api`(NestJS) + `apps/web`(React+Vite)가 빈 상태로 준비돼 있어, `pnpm install && pnpm dev`로 바로 부팅됩니다. 이 위에 도메인을 쌓으세요. DB/ORM/인증 라이브러리는 직접 추가하면 됩니다. (자세한 건 아래 **6장 스타터 사용법**)

### 백엔드
- **NestJS 11** — controller / service / repository 3-tier 구조 권장
- **Drizzle ORM** — 스키마 정의 + 마이그레이션
- **DB**: SQLite (`better-sqlite3`)
- **`@nestjs/swagger` (필수)** — API 문서를 작성해 주세요

### 프론트엔드
- **React 19** + **TypeScript 5.7** + **Vite**
- **라우팅: TanStack Router** — 보호 라우트(인증 가드)를 이걸로 구현해 주세요
- **UI: [shadcn-admin](https://shadcn-admin.netlify.app/) 기반으로 구현** — 이 어드민 템플릿([저장소](https://github.com/satnaing/shadcn-admin))의 레이아웃/컴포넌트(사이드바, 테이블, 폼, 다이얼로그 등)를 활용해 화면을 구성해 주세요. shadcn/ui + Tailwind 조합입니다.
- 상태관리: **zustand** 권장 (자유)
- 서버 상태/데이터 패칭: TanStack Query 또는 자유

> **코드 스타일은 `biome.json`(저희 실무 설정)을 따릅니다.** 제출 전 `pnpm lint`로 확인해 주세요. (설정은 아래 스타터 사용법 참고)

## 5. 제출 방법

이 레포를 **본인의 private 레포로 복제**한 뒤, 브랜치를 만들어 작업하고 **Pull Request**로 제출합니다. (다른 지원자와 코드가 섞이지 않도록 반드시 **private**으로 만들어 주세요.)

1. 이 레포를 본인 계정/조직의 **새 private 레포**로 올립니다.
   ```bash
   # 이 레포를 clone 받은 뒤, 본인의 새 private 레포를 origin으로 연결
   git remote remove origin
   git remote add origin https://github.com/<본인>/<레포명>.git   # 미리 만든 private 레포
   git push -u origin main
   ```
2. 본인 레포 **Settings → Collaborators**에서 리뷰어 **`TW-Tech-Reporter`** 를 collaborator로 초대해 주세요. (초대해야 PR 리뷰가 가능합니다)
3. **작업용 브랜치**를 만들어 그 위에서 구현합니다. (`main`에 직접 커밋하지 말고)
   ```bash
   git checkout -b feat/assignment
   ```
4. 작업이 끝나면 `feat/assignment` → `main` 으로 **Pull Request**를 생성하고, **Reviewers에 `TW-Tech-Reporter`** 를 지정해 주세요.
   - PR 제목은 **`이름 - 과제 제출`** 형식으로 적어 주세요. (예: `홍길동 - 과제 제출`)
5. PR을 열면 **제출 양식이 본문에 자동으로 채워집니다.** ([`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md)) 각 항목을 채워 주세요 — 실행 방법, 테스트 계정, 구현 범위, 기술 선택과 그 이유, 가정, AI 도구 사용. **무엇을 만들었는지만큼 "왜 그렇게 판단했는지"를 보고 싶습니다.**
6. PR 링크를 회신 메일로 보내 주세요.

### 제출 기한
- **영업일 기준 3일** — 과제 안내를 받은 날로부터. 정확한 마감 일시는 채용 담당자가 별도 안내

### 환경 요건
- Node.js 20 이상 (가능하면 22 LTS)
- `git clone` 후 README의 명령만으로 로컬에서 실행되어야 합니다. (`.env`가 필요하면 `.env.example` 제공)
- **시드 데이터 필수**: 최소한 ADMIN 1명 + USER 1명, 역할 2개, 메뉴 몇 개와 역할-메뉴 매핑이 들어 있어야 권한별 메뉴 차이를 바로 확인할 수 있습니다.
- **SQLite DB 파일을 함께 커밋해 제출해 주세요.** (예: `apps/api/data.db`) 시드까지 채워진 상태로 올려 주시면, 받는 즉시 실행해 확인할 수 있습니다. 별도 마이그레이션/시드 명령이 필요하다면 그 절차도 PR 양식의 "실행 방법"에 적어 주세요.

## 6. 스타터 사용법

이 레포는 **모노레포 골격만** 제공합니다. 실제 도메인 구현은 여러분의 몫입니다.

> ⚠️ 의도적으로 `node_modules` 와 lock 파일은 포함하지 않았습니다. 아래대로 직접 설치하세요.

### 구조
```
.
├── package.json            # 루트 — turbo 스크립트
├── turbo.json              # 빌드 파이프라인
├── pnpm-workspace.yaml     # 워크스페이스 (apps/*)
├── tsconfig.base.json      # 공통 TS 설정
└── apps/
    ├── api/                # 백엔드 (NestJS) — health 엔드포인트만 있는 빈 상태
    └── web/                # 프론트 (React + Vite) — 빈 화면만 있는 상태
```

### 시작하기
```bash
# 1) 의존성 설치 (pnpm 권장)
pnpm install

# 2) 개발 서버 (api + web 동시)
pnpm dev
#    개별 실행
pnpm dev:api     # http://localhost:3000  (GET /health 로 확인)
pnpm dev:web     # http://localhost:5173

# 3) 빌드 / 타입체크
pnpm build
pnpm type-check

# 4) 코드 스타일 (Biome — 우리 실무 설정 그대로 포함)
pnpm lint         # 검사만 (biome check .)
pnpm lint:fix     # 자동 수정 (biome check --write .)
pnpm format       # 포맷만 (biome format --write .)
```

- 프론트의 `/api/*` 요청은 `apps/web/vite.config.ts` 프록시를 통해 백엔드(3000)로 전달됩니다.
- 스타터의 `apps/web`은 **순수 React + Vite 빈 상태**입니다. **Tailwind · shadcn/ui · TanStack Router는 직접 설치·셋업**해 주세요. (shadcn-admin은 이 조합 위에 구성됩니다 — 3장 참고)
- DB(SQLite), ORM(Drizzle 등), 인증 라이브러리도 **직접 추가**하세요.
- **코드 스타일은 `biome.json`에 저희 실무 설정이 그대로 들어 있습니다** (single quote, 세미콜론 최소, trailing comma, 줄 너비 100, 2칸 들여쓰기 등). 제출 전 `pnpm lint`로 확인해 주세요. VS Code는 `.vscode/settings.json`이 포함돼 있어, [Biome 확장](https://marketplace.visualstudio.com/items?itemName=biomejs.biome)만 설치하면 **저장 시 자동 포맷·import 정리**가 됩니다.
- 패키지 구조·라이브러리 선택은 자유지만, **루트에서 `pnpm dev` 한 번으로 api·web이 함께 뜨는 구조**는 가능하면 유지해 주세요. 실행하기 편합니다.

## 7. 자주 묻는 질문

**Q. 경력 무관인데, 어디까지 구현해야 하나요?**
1장의 기능을 완성하는 것이 기본입니다. 그 외는 필요하다고 판단되는 만큼 하면 됩니다. 면접에서는 작성한 코드와 그 판단을 묻습니다.

**Q. AI 도구(Copilot, ChatGPT, Claude 등)를 써도 되나요?**
네, 자유롭게 사용하세요. 다만 **본인이 모든 코드를 설명할 수 있어야** 합니다. 면접에서 코드에 대해 질문합니다.

**Q. 권한을 어디까지 구현해야 하나요?**
**역할(Role) 단위로 구현하면 됩니다.** 사용자에게 역할을 주고, 역할에 따라 메뉴 접근이 갈리면 됩니다.

**Q. 메뉴 필터링은 프론트에서 하나요 백엔드에서 하나요?**
**둘 다 괜찮습니다.** 어떤 방식을 왜 골랐는지 PR 양식에 적어 주세요. 다만 권한 판단 자체는 서버에서 이뤄지도록 해 주세요.

**Q. UI는 꼭 shadcn-admin으로 해야 하나요?**
네, **[shadcn-admin](https://shadcn-admin.netlify.app/) 기반으로 구현**해 주세요. 이 템플릿의 레이아웃·컴포넌트를 활용하면 됩니다. 라우팅은 **TanStack Router**를 사용해 주세요.

**Q. 디자인을 어디까지 신경 써야 하나요?**
픽셀 단위 완성도는 보지 않습니다. **기능이 명확히 동작하고 상태(로딩/에러/빈 목록)가 처리**되면 됩니다.

**Q. 막혔을 때 편하게 질문해도 되나요?**
네, 4장의 오픈채팅방에서 **편하게 물어보세요.** 혼자 시간을 허비하는 것보다 묻는 게 낫습니다.

좋은 결과 기대하겠습니다. 화이팅입니다! 🙌
